# Runeclicker

Azure Static Web Apps project for the [Runeclicker](https://runeclicker.kmroz.com/) game by [Krzysztof Mrozinski](https://kmroz.com/).

---

## Project Structure

```text
.github/
  workflows/
    azure-static-web-apps.yml   # CI/CD pipeline for app content
    terraform.yml               # Manual Azure infrastructure pipeline
infra/
  terraform/                    # Azure resource group and Static Web App IaC
public/
  hacktimer/                    # Static worker/runtime assets
  images/                       # Static game images
  tables/                       # Static game data JSON
src/
  components/                   # React UI components
  logic/                        # Game state and data-loading logic
  types/                        # Game data types
index.html                      # Vite entry HTML
package.json                    # npm scripts and dependencies
staticwebapp.config.json        # Azure Static Web Apps configuration
vite.config.ts                  # Vite configuration
```

The `dist/` directory is generated by Vite during the build process and is not committed to the repository. The `public/` directory is source content for the Vite build and is committed.

## Azure Infrastructure

Terraform configuration lives in `infra/terraform` and provisions:

```text
Resource group
└── Static Web App
```

Terraform is intended to run from GitHub Actions only:

- Manual workflow runs can plan, apply, destroy, and bootstrap the remote Terraform state backend.

The Terraform state storage account is a separate GitHub Actions backend dependency.

Terraform does not run automatically for pull requests or pushes to `main`. The normal PR and merge path is reserved for updating the existing Azure Static Web App through `.github/workflows/azure-static-web-apps.yml`.

To test a new resource group and Static Web App without touching the production state, run the `Terraform` workflow manually with:

| Input | Example |
| --- | --- |
| `operation` | `apply` |
| `bootstrap_state` | `true` on the first run only |
| `resource_group_name` | `rg-runeclicker-app-sbx` |
| `static_web_app_name` | `swa-runeclicker-app-sbx` |
| `state_key` | `runeclicker/sbx/app.tfstate` |

Use the same `state_key` and names with `operation=destroy` to remove the sandbox resources.

Required GitHub variables:

| Variable | Purpose |
| --- | --- |
| `AZURE_CLIENT_ID` | Azure identity used by GitHub Actions OIDC |
| `AZURE_TENANT_ID` | Azure tenant ID |
| `AZURE_SUBSCRIPTION_ID` | Azure subscription ID |
| `TF_STATE_STORAGE_ACCOUNT_NAME` | Azure Storage account for Terraform state |

Optional GitHub variables:

| Variable | Default |
| --- | --- |
| `TF_STATE_RESOURCE_GROUP_NAME` | `rg-runeclicker-state-shared` |
| `TF_STATE_CONTAINER_NAME` | `tfstate` |
| `TF_STATE_LOCATION` | `eastus` |

Keep `AZURE_STATIC_WEB_APPS_API_TOKEN` pointed at the existing production Static Web App. Do not replace it with a sandbox deployment token unless you intentionally want the deployment workflow to target that sandbox app.

## Deployment Workflow

The app is deployed via `.github/workflows/azure-static-web-apps.yml`.

| Setting | Reason |
| --- | --- |
| `npm ci` | Installs exactly from `package-lock.json` |
| `npm run lint` | Checks the React and TypeScript source before publishing |
| `npm run build` | Builds the production Vite output |
| `app_location: dist` | Deploys Vite output only |
| `skip_app_build: true` | Prevents Azure from trying to build the app again |

Pull requests targeting `main` create an Azure Static Web Apps preview environment for app changes. Pushes to `main` build and deploy those changes to the existing Azure Static Web App.

## Azure Static Web Apps Configuration

`staticwebapp.config.json` controls Azure Static Web Apps behavior:

- Rewrites client-side routes to `index.html`
- Excludes static assets and game data from the navigation fallback
- Sets `X-Content-Type-Options` and `Referrer-Policy` security headers globally

The workflow copies this file into `dist/` after the Vite build.

## Required GitHub Secret

The deployment workflow requires one GitHub Actions secret:

```text
AZURE_STATIC_WEB_APPS_API_TOKEN
```

To retrieve the token:

1. Azure Portal -> Static Web App -> Manage deployment token

To add it to the repository:

1. GitHub repository -> Settings -> Secrets and variables -> Actions -> New repository secret
2. Name: `AZURE_STATIC_WEB_APPS_API_TOKEN`
3. Value: deployment token from Azure

The same token is also available as the sensitive Terraform output `deployment_token` after the Static Web App has been applied. Treat Terraform state as sensitive because it contains this token.
